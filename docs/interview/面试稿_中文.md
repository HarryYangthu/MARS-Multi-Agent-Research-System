## 0. 一句话电梯陈述(背熟)

> 我做了一个**研究型多 Agent 系统 MARS**:把"研究问题 → 文献调研 → 假设 → 实验设计 → 编码 → 仿真 → 论文"这条科研链路,拆成 **5 个领域 Agent + 1 个对话式主控 Agent**,用一套 **Schema 治理 + 人在环 (HITL) + 自愈反馈环路**的底座串起来,让研究员用一句自然语言就能驱动从想法到论文的完整闭环。全栈实现,约 1.7 万行代码,后端 `mypy --strict` 零错误、依赖方向 CI 强制、200+ 单测。首个落地场景是我自己的研究方向 —— **FDD Massive MIMO 的无源互调 (PIM) 对消**。

## 1. 项目背景:为什么做这个

**痛点**:我自己是做无线信号处理(PIM 对消)的算法研究员。一个研究周期(调研→设计实验→改代码→跑仿真→分析→写论文)往往要几周到几个月,且大量重复劳动:查文献、配实验矩阵、改 baseline 代码、跑消融、整理结果。

**判断**:LLM Agent 已经能胜任其中很多环节,但市面上的 AutoGPT/通用 Agent 有三个致命问题,正好是科研场景不能容忍的:
1. **不可控**:黑盒跑一长串,中间产物没法审查/干预;
2. **不可信**:输出格式漂移,下游没法消费;
3. **不可复现**:跑完没有完整沉淀,无法回放、无法追责。

**我的目标**:不是再做一个"更聪明的 Agent",而是做一个**让 Agent 在科研场景下可控、可信、可复现**的**底座 (harness)**。把"研究周期从月级压到周级"作为北极星指标。

## 3. 攻克的核心难题(面试重点 — 每个都是一个"故事")

### 难题 1:如何让 LLM 输出"可被下游程序消费"——Schema 是脊柱

**问题**:LLM 输出是自由文本,5 个 Agent 串成流水线时,上游产物下游根本没法可靠解析。

**我的方案**:定义 **"Markdown body + YAML frontmatter"** 的统一产物格式,frontmatter 必须通过对应的 **JSON Schema** 校验(`proposal.v1`/`experiment_plan.v1`/`code_spec.v1`/`run_log.v1`/`report.v1`/`diagnosis.v1`,共 6 种)。**核心原则:人手写的 md 和 Agent 生成的 md 对下游完全等价**,只要 Schema 过。

**技术细节**:
- `harness/schema/validator.py` 返回**结构化错误列表** `[{path, message}]`,前端能精准高亮缺失字段;
- 处理了 YAML 自动把 ISO 时间戳解析成 datetime 对象、而 JSON Schema 期望 string 的坑(写了 `_to_jsonable` 归一化);
- DeepSeek 常把文档包在 ```markdown 围栏里或加前言,我写了 `_unwrap_llm_text` 自动剥离;
- **把对应 Schema 的合规模板嵌进 system prompt** 当 few-shot 范例,首次合规率 ≥95%。

**为什么有亮点**:这是把"软件工程的契约思想 (contract)"引入 LLM 流水线 —— 用 Schema 当 Agent 之间的 ABI,解耦了"谁生产"和"谁消费"。

### 难题 2:Agent 跑完直接进下一步,没法审查——人在环 (HITL) 双层机制

**问题(我真实踩过的坑)**:最初 LLM 输出 schema 不合规时,被静默丢弃,orchestrator 检测到"没产物"就自动跳过进了下一节点 —— 表现就是"Agent 跑完直接下一个,什么都没让我审"。

**我的方案**:**两层 HITL**:
1. **每 Agent 输出**都进 review session(高频):人可以 edit / approve / reject,产物版本化 `v1/v2/approved`;
2. **5 个系统 Gate**(稀疏)强制阻塞关键节点。

**修复那个坑的三处改动**(能讲出 debug 过程是加分项):
- agent_runner:schema 不过**也要落盘 v1**(把错误当注释附在末尾),让 HITL 能看到;
- orchestrator:检测到"无产物"改为 **FAILED 中止**,而不是静默跳过;
- 把审批做成显著的紫色横幅 + 大号 Approve 按钮,顶部状态栏 `🔔 待审核 N` 脉动徽章。

### 难题 3:Gate 5 不能写成流程节点——挂在工具分发路径上

**问题**:有个 Gate(baseline_compatibility)要在"任何 Agent 调用会破坏 baseline 的工具时"触发,它不是流程的某个 checkpoint,而是横切所有工具调用。

**我的方案**:把 Gate 5 **hook 进 `harness/tools/registry.py` 的 `dispatch()` 路径** —— 每次工具调用前先过 Gate 检查,读项目 `AGENTS.md` 的 protected_paths + 正则匹配(比如保护 `forward(x, stream_label)` 接口签名、`baseline/` 目录、生产代码类),违反就阻塞。这是**面向切面 (AOP)** 的思路,而不是把检查散落在每个流程节点里。

### 难题 4:执行结果不达预期怎么办——自愈反馈环路

**这是最有"研究系统"味道的部分。**

**问题**:跑出来 RES 指标没达标,人工要去判断"是 coding 写错了,还是实验参数配错了",再回去改、重跑。

**我的方案**:**闭环自愈**:
- Execution 跑完 → **diagnosis 节点**:规则引擎 (`diagnostics.py`) 做**自动追责** —— 三个分析器:`metrics_gap`(指标差距)、`config_sanity`(实验配置问题→怀疑 experiment)、`code_change_risk`(代码风险→怀疑 coding);
- `BridgeAgent` 决策拉回哪个阶段 + 预算控制(`max_iterations`);
- **orchestrator 动态往 RunGraph 加节点链**:`coding#2 → execution#2 → diagnosis#2 → writing`,attempt+1,形成循环,直到达标或耗尽预算。

**RunGraph 是通用 DAG,不写死线性拓扑** —— 线性顺序写在 `bridge/workflow_service.py`,运行时能动态改图,这是自愈循环能实现的关键。

### 难题 5:让研究员一句话驱动全流程——Commander 对话式主控 + 双层状态机

**这是我最近做的、最能体现 Agent 编排能力的部分。**

**问题**:前面 5 个 Agent 是流水线,但用户还是要手动选入口、填表单。我想要一个能对话的"指挥官"。

**我的方案**:**Commander 主控 Agent**(DeepSeek 驱动),坐在 Bridge 层:
- **双层状态机咬合**:上层 **7 态会话 FSM**(idle→clarifying→planning→awaiting_confirm→executing→awaiting_review→reporting),下层复用现有 pipeline 节点状态机;
- **意图路由**:用户说"我已经有想法了,直接验证" → Commander 理解后**跳过 Idea Agent,直接从 Experiment 入口启动**;
- **工具调用 (function-calling)**:我没用 provider 原生 tool-use,而是让 LLM 输出**严格 JSON 决策** `{reply, next_state, actions}`,自己跑 **ReAct 循环**(执行工具→喂回结果→再决策)。好处:对所有 provider 通用、mock 也能跑、更可控;
- **不重写引擎**:Commander 是"大脑",通过工具复用现有 orchestrator + 自愈引擎当"手脚",职责清晰。

**实测**:一句"帮我跑 ATK-MoE 路由简化的消融,对比不同 expert_count 对 RES 的影响,走到论文",Commander **4 分钟自动跑完** idea(debate)→experiment(自主设计 memory×lr 消融网格)→coding→execution(真实 PIM 仿真)→writing(debate),5 阶段全 done。

### 难题 6:零依赖也能演示——三重 Mock 兜底

**问题**:面试/CI/没 GPU 没 API key 的机器上,系统也得能跑完整 demo。

**我的方案**:三处自动降级:
- `mock_provider`:没 API key 时返回**Schema 合规的占位**(不是随便的文本,是能过校验的);
- `mock_simulation`:没 GPU 时合成 loss 曲线;
- `mock_debate`:provider 不够时降级。
- **自动降级逻辑**:debate 根据可用 provider 在 `real_multi_model / single_model_simulated / mock_debate` 三模式间切换。

这保证了**零外部依赖下完整 11 步 demo 能跑通**,是 V0 的硬验收指标。

## 4. 真实物理仿真(领域深度 — 区别于"套壳 Agent"的关键)

面试官可能质疑"你这就是调 API 套壳"。我的回答:**Execution Agent 跑的是真实物理仿真**。

**我做了什么**:为我的 PIM 研究方向写了一个**物理真实但 CPU 秒级**的双载波 PIM 对消仿真(`execution/pim_cancellation.py`):
1. **双载波信号**:`x = A1·e^{j2πf1n} + A2·e^{j2πf2n}`,3 万点复数基带;
2. **奇阶交调 PIM**(带记忆效应):memory polynomial 非线性,3 阶交调落在 2f1−f2;
3. **对消器**:memory-polynomial,用梯度下降拟合;
4. **关键工程坑**:双载波 memory-polynomial 基函数高度共线、Gram 矩阵病态,普通梯度下降**直接发散(loss 爆到 1e86)**。我用 **QR 正交化**把基函数正交化,在正交空间梯度下降,保证良态、单调收敛;
5. **物理正确性**:RES 随对消器记忆深度真实区分 —— memory=2 → −15dB(压不干净),memory=8 → −30dB(到噪声地板)。**记忆深度不足导致对消不全,这正是 PIM 对消的真实物理规律**。

**为什么重要**:这证明系统不是"嘴上跑实验",而是真的有领域物理在里面,且 Experiment Agent 能**自主设计** memory×lr 消融网格,Execution 真实跑出有区分度的 RES。同时留好了钩子:上真实 4×L40S + 真实 `.mat` 数据时,把这个分支换成 subprocess 调真实 7 层模型训练即可。

## 5. 用到的前沿技术 / 自己的创新

### 前沿技术(业界在用的)
- **Multi-Agent Orchestration**:5 领域 Agent + 主控,类似 Anthropic 的 multi-agent research、AutoGen/CrewAI 的编排思路;
- **RAG / 知识库**:4 区独立知识库(文献/方法/代码/run档案),语义检索 + 沉淀闭环;
- **Multi-model Debate**:3 模型 × 多轮辩论 + Critic 综合(用 DeepSeek-V3 chat 当 proposer、DeepSeek-R1 reasoner 当 critic,跨模型版本);
- **Function-calling / ReAct**:Commander 的工具调用循环;
- **HITL (Human-in-the-loop)**:每节点审查 + 系统 Gate;
- **LLM 多 backend 抽象**:Anthropic/OpenAI/Qwen/Gemini/DeepSeek/本地 vLLM 统一接口。

### 我自己的创新点(讲这些)
1. **Schema-as-Spine(Schema 当脊柱)**:把契约思想引入 Agent 流水线,"人写 md = Agent 写 md",彻底解耦生产与消费;
2. **Gate 5 切面式 baseline 保护**:不在流程节点,而是 hook 工具分发路径,横切所有工具调用;
3. **规则追责 + 动态拉回的自愈环路**:Execution 不达标 → 自动诊断锅在哪 → 动态改 DAG 拉回重跑 → 预算控制;
4. **双层状态机 + 对话主控**:会话 FSM 与 pipeline FSM 咬合,Commander 用 JSON-ReAct 复用而非重写引擎;
5. **三重 Mock 兜底 + Schema 合规的 mock**:零依赖可演示,mock 产物也能过 Schema 校验,下游无法区分真假;
6. **QR 正交化解病态 PIM 仿真**:把数值线代知识用在 Agent 系统的执行后端。

## 6. 工程能力(展示软件工程素养)

- **架构纪律**:5 层依赖**单向**,用 `import-linter` 4 条 contract **CI 强制**(harness 禁止 import 上层、bridge 禁止直接 import 具体 Agent 走注册表反转依赖、agents 禁止 import bridge/api、整体分层)。这是很多人忽略的——**用工具把架构约束变成可执行的 CI 检查**;
- **类型安全**:`mypy --strict` 全量零错误(125 文件);
- **测试**:200+ 测试,覆盖 schema 合规率(≥95%)、5 个 Gate 触发、baseline 匹配召回/精度、多实验并发、端到端 demo;
- **可观测性**:每次 run 完整沉淀到 `runs/<id>/` 9 个子目录(input/context/idea/.../events),完全可回放/可审计/可追责,还有 TraceRecorder 做 span 追踪;
- **依赖反转**:Bridge 通过 `agent_registry` 反查 Agent(Protocol 结构化类型),不直接依赖实现;
- **优雅降级**:API 失败重试 + mock 兜底,不损坏 run 状态;
- **端到端验证**:写了 `acceptance.sh` 一键验收 + `run_demo.py` 11 步主脚本。

## 7. 对业界的理解(展示判断力)

- **为什么 Agent 落地难**:不是模型不够强,是**可控性、可信度、可复现性**不够。MARS 的设计就是冲着这三点去的(Schema/HITL/沉淀)。
- **为什么垂直比通用更可能落地**:通用 Agent 没有领域约束(baseline 保护、领域指标、复用资产),容易跑飞;垂直场景能把领域知识编码进 Gate 和 Schema。
- **Harness 与 Agent 解耦的价值**:harness 不知道任何具体 Agent,换 Agent/换项目/换部署都不动 —— 这是平台思维,对应"AI Infra"的价值主张。
- **HITL 不是退步而是必需**:在高价值、低容错的科研/生产场景,全自动反而危险,人机协同 (human-AI teaming) 是更现实的形态。
- **对标**:概念上接近 Google 的 "AI Co-scientist"、Stanford 的科研 Agent、Sakana 的 AI Scientist,但我更强调**工程可控性**而非"全自动发论文"。

## 8. 可能被问到的问题 + 参考回答

**Q1:这不就是调 LLM API 套壳吗?**
A:三点区分:① 真实物理仿真(QR 正交化的双载波 PIM 对消,不是假数据);② Schema 治理 + 自愈环路这类**机制创新**不在模型里,是系统设计;③ 架构纪律(import-linter/mypy strict/200 测试)是工程实力。套壳做不到自愈循环和零依赖可复现。

**Q2:那个 7 层真实模型其实没训练,是不是造假?**
A:我诚实说明:完整 7 层模型需要真 GPU + 真 `.mat` 数据,CPU 跑不动。所以 Execution 跑的是**物理真实的轻量替身**(双载波 PIM + memory-polynomial 对消),RES 随记忆深度真实变化,物理规律正确。钩子已留好,上硬件把分支换成 subprocess 调真模型即可。**我把"能演示的真实"和"需要硬件的真实"分清楚了,这本身是工程判断**。

**Q3:多 Agent 比单 Agent 强在哪?会不会过度设计?**
A:科研链路天然分阶段、每阶段需要不同的工具/模型/审查粒度(比如 Coding 要 baseline 检查,Execution 要并发调度),拆分让每个 Agent 职责单一、可独立审查/复用/沉淀。而且 debate 需要多模型视角。但我也克制:harness 是 agent-agnostic 的,没把拓扑写死,单 Agent standalone 也支持。

**Q4:怎么保证 LLM 输出稳定?**
A:Schema 校验 + system prompt 嵌模板 few-shot + 输出剥离围栏 + 失败落盘让人改,首次合规率 ≥95%,不过的进 HITL 补全。**我不假设 LLM 一定对,而是用机制兜住它的不确定性**。

**Q5:状态机为什么要两层?**
A:会话节奏(在澄清?在等确认?在汇报?)和流水线执行(节点 running/done)是两个正交的关注点。会话 FSM 管人机交互,pipeline FSM 管 Agent 执行,Commander 当桥梁。混在一起会耦合。

**Q6:并发/扩展性怎么做?**
A:V0 用 asyncio + 信号量(执行池并发上限 6),事件用 in-process bus(可切 Redis pub/sub),WebSocket per-run/per-experiment 频道隔离。V0 单用户单机,多用户/分布式是 V1(已留 project_isolation 接口)。

**Q7:如果让你重做,会改什么?**
A:① Commander 的自愈联动可以更深(目前追责是规则引擎 + LLM 解释,可升级为 LLM 主导);② 知识库的 embedding 现在是确定性 hash(为了零依赖),上线换 sentence-transformers;③ 前端组件多处独立轮询,该合并成单一状态订阅。

**Q8:这个项目花了多久?一个人做的?**
A:一个人全栈从零做的。架构设计先行(5 份顶层设计文档),然后严格按 7 个 Phase 端到端推进,每个 Phase 完成保证系统仍能跑通(end-to-end first,避免横向铺开最后集成失败)。

## 9. 项目水平自评(诚实版)

**优势**:
- **全栈深度**:1.7 万行、前后端 + Agent 编排 + 物理仿真,广度和深度兼具;
- **机制创新**:Schema 脊柱、切面 Gate、自愈环路、双层状态机,这些是**系统设计层面的原创**,不是调包;
- **工程纪律**:import-linter / mypy strict / 200 测试 / 完整沉淀,达到准生产水准;
- **领域结合**:真实 PIM 物理 + 真实研究代码/论文入库,不是 toy demo。

**坦诚的局限**:
- V0 单用户、未上真实 GPU 训练(物理替身);
- posttrain 是占位(V1);
- 部分能力 mock 兜底;
- 单人开发,某些模块深度 vs 广度有取舍。

**定位**:这是一个**准生产级的研究 Agent 平台 MVP**,体现了**系统架构 + LLM 工程 + Agent 编排 + 领域物理 + 全栈**的综合能力。对标岗位:**AI Agent / LLM 应用工程、AI Infra、研究工程师 (Research Engineer)**。如果对方是做 Agent 平台/科研 Agent/AI for Science 方向,这个项目高度契合。

**一句话**:我做的不是"又一个 Agent",而是"让 Agent 在科研场景可控、可信、可复现的底座",并用我自己的 PIM 研究方向做了真实落地验证。
