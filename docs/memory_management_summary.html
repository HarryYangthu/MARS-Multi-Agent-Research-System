保留什么

保留 mock-first 可降级、per-run 落盘可 replay/audit、四库分区清晰、KB API 可替换这些强项。

先补什么

先补 eval 门控写入、mock 隔离、冲突消解和遗忘治理。它们是正确性问题，不是后端性能问题。

为什么急

未门控、混 mock 的长期记忆会成为跨会话投毒通道；embedding 只会让污染库搜得更快。

当前 Memory 表面

系统没有单一 MemoryManager，而是由短期上下文、长期 KB、会话、Agent 自上下文和工作台记忆共同组成。

短期：Run / ContextPack

runs/<id>/ 保存 input、context、Agent 产物、HITL、events。BaseAgent.ContextPack 是实际喂给 LLM 的 working memory。

短期：Commander 会话

conversations/<conv_id>/ 保存 session 状态、关联 run、messages 和工具结果。当前主要是全量 replay。

长期：四库 KB

literature、methodology、code_assets、run_archive 是跨会话长期知识区。

长期：Agent 自上下文

docs/prompts/examples/evals/uploads 目前作为 Agent 长期自上下文。v2 将它们识别为程序/情景记忆。

长期：Coding workspace items

Coding workspace 维护手工 memory items 和 KB memory items。v2 建议收编进统一 MemoryRecord。

审计：Context Manifest

context_pack.vN.json 与 context_snapshot.vN.md 记录上下文审计。v2 要求与实际 LLM messages 同源。

v2 门控 Sedimentation 流程

1. Artifact 完成 Agent 输出 markdown + YAML frontmatter。
2. Schema approved? 现有 schema 校验与 HITL approval。
3. Eval passed? faithfulness、显著性、引用、provenance gate。
4. Mock quarantine is_mock=true 写隔离区，主召回与 baseline match 默认排除。
5. Conflict check content hash 去重、source upsert、语义 supersede/merge。
6. Write memory 写入 zone + memory_type，带完整 provenance 和生命周期字段。

统一记忆模型

存多久和是哪类认知要分开建模。四库解决存储分区，memory_type 解决召回和自进化 lever。

语义 Semantic 情景 Episodic 程序 Procedural
短期 ContextPack 中注入的事实和发现。 当前 run 轨迹、事件、HITL 反馈。 任务约束、当前 prompt、工具观察。
长期 四库里已验证 findings、baseline profile。 结构化 run episode、Agent examples、失败案例。 methodology、code assets、Agent prompts、eval scorecard。

MemoryRecord v2

v2 的关键是让每条记录自带治理信息，而不是只有 text + embedding。

memory_type is_mock confidence eval_status source_path content_hash supersedes salience ttl_days

记录角色

  • 门控类：eval_status.passed、is_mock、approved。
  • 召回类：memory_type、summary、confidence。
  • 生命周期类：supersedes、salience、ttl_days。
  • 溯源类：source_path、run_id、agent、content_hash。

冲突消解与版本

手段 处理 实现
去重 精确重复 content_hash
Upsert 同 source 重写 delete_by_source + write
Supersede 同实体新结论取代旧结论 新记录置 supersedes，旧记录置 superseded_by
Merge 互补近重复合并 后台 consolidation job 保留并集证据

遗忘与生命周期

手段 对象 规则
TTL 时效 run metrics / fingerprint ttl_days 到期失效或归档
衰减 久未召回或确认的记录 confidence / salience 随时间下降
剪枝 低价值记录 低于 salience 阈值进入剪枝候选
Mock 强 TTL quarantine 区 短 TTL，demo 后清理，排除主召回

MemorySelector v2

召回不只看 similarity。v2 先 filter，再综合相似度、recency、confidence、eval_status、salience，加 rerank，最后注入 summary 而非全文。

filter(zone, project, memory_type, not superseded, not mock)
  -> score(similarity, recency, confidence, eval_status, salience)
  -> rerank
  -> top-k
  -> inject summary

Context Compiler 同源

一次 compile 同时生成真实 LLM messages 和 manifest，解决当前“manifest 不完全等于实际输入”的偏差。上游 artifact 不再整篇 dump，而是蒸馏为关键结论 + 引用指针。

ContextPack.compile()
  -> messages  # 真正发给模型
  -> manifest  # 同一结构投影出的审计记录

Agent 自上下文接自进化

prompts/examples/evals/uploads 不再只是文件夹，而是可治理、可召回、可优化的程序/情景记忆。

自上下文 认知类型 自进化用途
prompts 程序记忆 prompt 优化对象，从 eval 反馈迭代。
examples 情景记忆 few-shot 来源，从高分且 coherent 的 run 中挑选。
evals 情景/程序 Agent 校准和自进化信号。
Coding workspace items 混合 收编进 MemoryRecord，消除平行表面。

Mock 隔离纪律

  • is_mock=true 写入 quarantine zone，不进四库主区。
  • research / hardware profile 默认排除 mock。
  • baseline match 绝不使用 mock fingerprint 作为 current baseline。
  • quarantine 强 TTL，demo 后清理。

Eval ↔ Memory

  • eval 是记忆写入 gate：不过 faithfulness / 显著性 / 引用，不进主语义库。
  • eval scorecard 沉淀为情景/程序记忆。
  • 自进化从 prompt、few-shot、KB findings 三个 lever 学习。
  • 记忆的鲁棒性直接影响自进化安全性。

分阶段落地

Phase 1

同源 Manifest

ContextPack 一次 compile 产出 messages + manifest，解决审计不等于实际输入。

Phase 2

MemoryRecord v2 Schema

新增 memory_type、is_mock、confidence、eval_status、supersedes、salience、ttl，兼容旧 KBRecord 读取。

Phase 3

门控 Sediment + Mock 隔离

schema -> eval-pass -> mock -> salience -> conflict。mock 进 quarantine，排除主召回和 baseline。

Phase 4

冲突消解 + 遗忘

resolver 做 supersede/merge，consolidate 做去重、衰减、TTL、清 mock；baseline-current 改 profile。

Phase 5

MemorySelector + Context Compiler

加权、type-aware、rerank、JIT 召回；上游 handoff 变蒸馏包；Commander 加滚动摘要和 budget。

Phase 6

后端替换

file store 迁移到 ChromaDB 或 sqlite-vss。前提是库已门控、已去 mock、已会遗忘。

Phase 7

接自进化

prompts、examples、findings 接成 prompt / few-shot / KB 三个 lever，eval 门控写入作为反投毒边界。

成功标准

  • 每条 MemoryRecord 带 memory_type、is_mock、confidence、eval_status、source_path。
  • manifest 与真实 LLM messages 同源，可据 manifest 精确重建模型输入。
  • research/hardware 下 eval 不过的发现不进主库，mock 永不进入主召回和 baseline match。
  • 矛盾结论被 supersede，主召回只返回未被取代记录。

一句话

当前 MARS 记忆是干净、可追溯、可降级的 append-only 知识堆；v2 补上 eval 门控写入、冲突消解、遗忘治理，并用统一模型把分散表面收编。先装这三块，再上 ChromaDB。
