MARS Memory 管理机制总览

当前系统的 Memory 不是一个单体服务，而是由短期上下文、Run 落盘、四库长期知识库、Commander 对话历史、Agent 自上下文共同组成。核心长期记忆就是四个知识分区：文献库、方法库、代码资产库、实验运行档案。

扫描日期：2026-06-17。数据来自 knowledge/*/_index.json。

记忆流向

1. 用户与 Commander 用户请求进入 Commander，对话写入 conversations/<conv_id>/。
2. Run 创建 每次任务创建 runs/<id>/，包含 input、context、Agent 产物、HITL、events。
3. Agent Working Memory Agent 装载 system、project、task、upstream artifact、自上下文和工具结果。
4. 产物与审核 产物以 v1/v2/approved 版本保存，HITL 记录进入 hitl/review_log.jsonl。
5. 沉淀到四库 Agent 完成后通过 sedimentation extractor 写入四个长期知识分区。

当前 49 条

主要来源：Execution Agent，Experiment Agent 可用它做 baseline match。

短期 Working Memory

Agent 实际给 LLM 的上下文由 BaseAgent.ContextPack 管理，包括 system、project、task、upstream 和 metadata。上游已批准产物会被读入 downstream Agent 的 upstream handoff。

Context Manifest

harness/context 负责写审计用 manifest：context_pack.vN.json 和 context_snapshot.vN.md。当前它主要记录“这次调用装载了什么”的审计视角，和实际 LLM messages 还没有完全统一。

Run Memory

每次 run 都落盘到 runs/<timestamp>_<task>/。其中 run_state.json 保存 RunGraph 状态，events/*.jsonl 保存状态变更和 websocket 事件，Agent 产物按版本保存。

Conversation 与 Agent Memory

Commander 对话写入 conversations/<conv_id>/messages.jsonl。每个 Agent 的 docs/prompts/examples/evals/uploads 是长期自上下文。Coding workspace 还维护手工 memory items 和 KB memory items。

机制 职责 主要文件

当前优点

  • 四库分区明确，能区分文献、方法、代码和实验档案。
  • 每次 run 都完整落盘，具备 replay 和 audit 基础。
  • mock 模式下无外部模型和 GPU 也能跑完整链路。
  • KB 后端虽然简单，但 API 表面已经接近可替换实现。

主要短板

  • 实际 LLM messages 和 Context Manifest 还没有完全同源。
  • KB 当前是 append-only 风格，缺少 content hash 去重和 upsert。
  • Commander 对话历史全量 replay，缺少滚动摘要和 token budget。
  • V0 使用 JSON 向量库，规模上来后需要迁移到 ChromaDB 或 sqlite-vss。

建议优化顺序

  1. 先把真实 LLM messages 写入 Manifest，解决“审计记录和实际输入不一致”。
  2. 给 KB 记录加统一 metadata：source_path、content_hash、run_id、agent、approved_only。
  3. 在 KBStores 增加 upsert、delete_by_source、metadata filter。
  4. 落地 MemorySelector：按相似度、同 project、approved、recency、importance 选择上下文。
  5. 实现 Commander 滚动摘要和长 artifact reference 压缩。
  6. 最后替换底层 JSON store 为 ChromaDB 或 sqlite-vss。
